application.yml の概要

application.yml はアプリケーションの動作に関わる設定を管理します。

パス 説明 必須
config/application.yml アプリケーション全体から参照できるファイル。 ◯
config/application_{hostname}.yml ホスト拡張ファイル。  

application.yml には開発者がカスタムパラメータを定義するべきではありません。カスタムパラメータは site.yml か、カスタム設定ファイル内に定義するべきです。

属性一覧

application.yml で定義可能なパラメータは次の通りです。

subdomain  

サブドメインをモジュール名に割り当てる。例えば foo.example.com というリクエストは foo モジュール、bar.example.com は bar モジュールを起動する。サブドメインが無い、またはサブドメインに該当するモジュールが見つからない場合はデフォルトモジュールを起動する。

extension .do (string)   アクションをリクエストする際に付加する拡張子。HelloAction に対する拡張子が ".do" の場合は "hello.do" がアクションパスとなる。"" (空文字) を指定することで拡張子を付けないよう設定することも可能。
response   HTTP レスポンスの動作を定義。
callback ob_gzhandler (string) レスポンスが出力される際にコールする関数。
※デフォルトでは "ob_gzhandler" が指定されているため、gz エンコードされたデータをクライアントに返す。尚、この機能を利用するに辺り、PHP の zlib 拡張モジュールが有効化されている必要がある。
※コールバックを無効にしたい場合は "none" を指定する。
session   HTTP セッションの動作を定義。
name PHPSESSID (string) セッション名。セッション ID をページ間で引き渡す際に使用します。
autoStart ◯ (bool) セッションの開始設定。セッションを使用しないアプリケーションでは FALSE を指定することで、アプリケーション全体のパフォーマンスが向上します。
encrypt (bool) session.encrypt 項を参照。
handler   (string) セッション開始時に実行するハンドラの指定。詳しくは delta の拡張 - セッションハンドラの利用 を参照。
cookieLifetime 0 (int) Cookie の有効期限。(単位は秒)
0 を指定時はブラウザを閉じるまで有効となる。
cookiePath / (string) Cookie の設定パス。
cookieDomain (string) Cookie の有効ドメイン。
cookieSecure (bool) SSL 通信時のみ Cookie を送信する場合は TRUE を指定。
cookieHttpOnly (bool) クライアントスクリプト (JavaScript 等) から Cookie へのアクセスを拒否する場合は TRUE を指定。
timeout -1 (int)

セッションの有効時間。(単位は秒) 最終アクセス時刻より 'timeout' 秒を超えたリクエストが発生した場合、クライアントが保持する全てのデータを破棄する。PHP が持つ 'session.gc_maxlifetime' と異なり、モジュール単位での制御が可能。この時、'session.gc_divisor'、'session.gc_probability' の値は考慮されない。
なお、0 以下の値が指定された場合はタイムアウト処理を行わない。 (PHP のガベージコレクションに依存する)
store transparent (string)
  • cookie: クライアントのセッション管理に Cookie を使用するよう強制する。セッション固定化攻撃に有効だが、Cookie をサポートしないエージェントの場合、セッション機能が働かないことになる。
  • transparent: クライアントが Cookie を受け付けない場合、セッション ID を URI に埋め込むよう制御する。
  • custom: セッションの保持方法をフレームワーク側で制御しない。(php.ini 等の設定に依存)
updateSpan -1 (int) セッション ID の自動更新間隔。(単位は秒)
セッション ID を定期的に更新することで、セッション固定化攻撃 (Session Fixation) の対策を行います。デフォルト (-1) では自動更新を行いません。'store' 属性が 'transparent' または 'custom' (セッション ID を URI に埋め込む設定) の場合、セッション ID の自動更新は無効となります。

domain     (array) テーマを適用するドメインとテーマを連想配列の形式で指定。ドメイン名は example.com 形式のほか、*.example.com 形式にも対応。("*." は example.com にもマッチする)
modules     (array) テーマを適用するモジュールリスト。"*" を指定した場合は全てのモジュールが対象となる。
extension     (array)

テーマ拡張ディレクトリ。指定したディレクトリもテーマの一部と見なす。指定可能な値は次の通り。

  • data
  • modules.{module_name}.behaviors
exception   例外発生時の動作を定義。詳しくは 例外のハンドリング を参照。
(+) type (例外クラス名) (string) ◯ 捕捉する例外クラス名。
delegate (例外を委譲するクラス名) (string) ◯ 捕捉した例外を委譲するクラス。(Delta_ExceptionDelegate の実装クラス)
continue (bool) TRUE を指定した場合、例外のハンドリング後も次のハンドラへ処理を引渡します。
error   エラー発生時の動作を定義。

output (bool) ◯ デバッグコードを出力するかどうかの設定。例外発生時のトレース表示や、Delta_DebugUtils::isDebug() 関数の動作に影響。セキュリティ上の観点から本番環境では FALSE を推奨。

php 項を参照。

autoload

Delta_ClassLoader が参照するクラスの自動読み込みパスを追加します。

YAML Code#
autoload:
  # {autoload_id}: {autoload_path} 形式。{autoload_id} はパスを一意に識別する ID
  plugin: /var/lib/plguins
  extension: /var/lib/extensions

上記の設定により、例えば plugins ディレクトリ下にある Example.php へのパスが追加され、アクションクラスからはパスを指定することなくクラスを呼び出すことが可能になります。

charset.default

アプリケーションのデフォルトエンコーディング形式を指定します。default で指定した形式は、内部・入力・出力エンコーディング (及び出力ヘッダ) にそれぞれ適用されます。また "auto" を指定することで、mbstring オプションによる個別のエンコーディング指定が可能となります。

YAML Code#
default:
  charset: auto

php:
  mbstring.internal_encoding: UTF-8
  mbstring.http_input: Shift_JIS
  mbstring.http_output: Shift_JIS

default に "auto" を指定する場合、mbstring.internal_encoding の指定は必須となります。オプションに "pass" や "auto"が指定されている場合は実行時にエラーが発生します。

php

php.ini の設定をアプリケーションレベルで上書きします。

YAML Code
php:
  error_log: <?php echo APP_ROOT_DIR . '/logs/fatal_log'; ?>
  display_errors: On
  max_execution_time: 30

モジュールやアクションが見つからない場合の動作

リクエストされたモジュールやアクションが見つからない場合、ルータは設定ファイルに定義されたデフォルトモジュール (アクション) の起動を試みます。ルータが実行するマッチング処理は下表を参照して下さい。

コンソールアプリケーションは対象外です。コンソールアプリケーションでアクションが見つからない場合はその場で例外が発生します。

  実行モジュール 実行アクション 結果

リクエスト URI が routes.yml のパターンにマッチする

  
  フォワード先のモジュールが指定されている  
    フォワード先のアクション名の解析に成功 リクエストモジュール リクエストアクション リクエストされたアクションを起動
    フォワード先のアクション名が不明 (URI にアクション名が含まれない)  
      "action.default" アクションが存在する リクエストモジュール "action.default" リクエストモジュールの default アクションを起動
      "action.default" アクションが存在しない  
        "action.unknown" アクションが存在する リクエストモジュール "action.unknown" リクエストモジュールの unknown アクションを起動
        "action.unknown" アクションが存在しない - - 例外が発生

    "action.entry" モジュールが存在する "action.entry" リクエストアクション "action.entry" モジュールのリクエストアクションが起動

        "module.unknown.action" アクションが存在しない - - 例外が発生
      "module.default.module" モジュールが存在しない - - 例外が発生
リクエスト URI が routes.yml のパターンにマッチしない ("module.unknown.module" モジュールを検索)
